Grant Home Assistant access?

Enabling this lets Claude read from and control your Home Assistant instance. With your token, Claude can:

  • Read every entity's state (lights, locks, climate, cameras, sensors, etc.)
  • Call any HA service — turn devices on/off, lock/unlock doors, change scenes, run scripts, fire automations
  • Read logbook, search entities, evaluate templates
  • Create, modify, and delete automations, scripts, dashboards, and helpers
  • Trigger Home Assistant backups

Scope is whatever your long-lived token can do in HA — admin tokens can do everything. Built-in safety hooks still apply on top: secrets and auth files are off-limits entirely, the installed-apps folder (/addons) is read-only, and destructive shell commands (rm -rf, force pushes, mkfs, etc.) are blocked.

You can revoke access at any time from HA → Profile → Security, or just turn this toggle off.

⚠ Make sure your Home Assistant instance is backed up before enabling. Claude is an AI and can make mistakes — automations, scripts, and dashboards can be modified or deleted, and there's no undo.

        Built-in security policy

        These are the built-in security controls regardless of what has been toggled on or off. They are hard-coded into the app, but AI can make mistakes — always keep your own backups of your Home Assistant config.

        Files Claude can never read or change

        • /config/secrets.yaml
        • Home Assistant auth and onboarding storage (.storage/auth*, .storage/onboarding*)
        • Anything matching *token*, *password*, *credential*, .env*
        • The Claude CLI's auth file (.claude.json) — Claude can't see its own credentials

        Files auto-backed-up before edit

        • configuration.yaml, automations.yaml, scripts.yaml, scenes.yaml
        • groups.yaml, customize.yaml, ui-lovelace.yaml
        • Lovelace storage files (.storage/lovelace*)

        Before any edit, the app snapshots a copy to <file>.bak.<timestamp>. Note: deletion via plain rm isn't path-blocked — it goes through the Bash permission flow.

        Commands Claude can never run

        • rm -rf in any form
        • git push --force, git reset --hard, git clean -f
        • ha core restart/stop, ha host reboot/shutdown
        • dd if=…, mkfs.*, wipefs
        • Anything invoking sudo or su -
        • Pipe-to-shell (curl … | sh, wget … | sh)

        These are blocked outright — no approval can override them. Claude has to take a different path.

        Network & privacy

        • WebFetch prompts for confirmation per domain (when the setting is on); outbound HTTP is not otherwise firewalled
        • No telemetry, no analytics, ever
        • Conversation history stays on this Home Assistant host
        • Images you attach are sent to Anthropic as part of the message and not pushed anywhere else

        Audit log

        Every tool call (Read, Edit, Write, Bash, WebFetch, etc.) — allowed and blocked — is appended to /config/claude-code-messages-audit.log with timestamp, tool name, arguments, and outcome. Append-only from the app; you can review or rotate it any time.

        What this does NOT cover

        • A destructive prompt you manually approve anyway
        • Plain rm of a protected file (only rm -rf is auto-blocked; recovery via the .bak.* snapshot)
        • Outbound HTTP to arbitrary domains if you've turned off WebFetch confirmation
        • Anthropic's handling of your data — see their privacy policy
        • Physical access to the HAOS host
        Project instructions

        Markdown appended to Claude's system prompt for every chat in this project. File lives at /config/claude-project-notes/… so you can also edit it directly.
